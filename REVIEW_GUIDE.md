# LaraVibe Repair Platform — Code Review Guide & Handover

Welcome to the LaraVibe Repair Platform codebase. This project is a complex, asynchronous AI orchestration engine that automatically repairs broken Laravel code by executing it in an isolated Docker sandbox, generating fixes via LLMs, and validating the repairs using Pest and mutation testing.

To save you time, you do not need to read the entire repository. The core logic of the platform is concentrated in the **15 files** listed below.

---

## 1. The Core Repair Loop (The Brain)
These files dictate the step-by-step logic of how a broken piece of code is analyzed, fixed, and tested.

- **`api/services/repair/orchestrator.py`**  
  *(Start Here)* This is the beating heart of the platform. It contains `run_repair_loop()`, which walks through the 13-step repair sequence (setup container → copy code → get AI context → AI pipeline → apply patches → pest test → mutation gate).
- **`api/services/repair/pipeline.py`**  
  Defines the multi-agent AI sequence: Planner → Verifier → Executor → Reviewer.
- **`api/services/escalation_service.py`**  
  The safety mechanism. If the AI gets stuck in a loop (repeated failures or identical patches), this file analyzes the history and injects "stern" prompts to force the AI to try a new strategy.

---

## 2. The AI & Prompt Engine
These files define how the platform communicates with LLMs (Claude, Gemini, DeepSeek, etc.) and how their responses are parsed.

- **`api/services/ai_service.py`**  
  Handles model rotation (falling back to different providers if one fails), builds the prompts, and safely extracts/parses XML responses from the LLMs.
- **`api/prompts/repair_prompt.md`** *(or equivalent prompt templates)*  
  The raw text instructions sent to the AI that enforce Spatie formatting, XML tagging, and Pest test generation.
- **`api/services/patch_service.py`**  
  The logic that takes the XML `<file>` tags generated by the AI and safely applies them as `full_replace` or `create_file` actions within the Docker container.

---

## 3. The Sandbox Environment (Docker & Laravel)
These files control the secure execution environment where the AI's PHP code is actually tested.

- **`api/services/sandbox_service.py`**  
  High-level Laravel operations. Contains the logic for detecting PHP namespaces, placing code in the correct `app/` directories, generating baseline Pest tests, and running the `pest --mutate` gatekeeper.
- **`api/services/docker_service.py`**  
  The low-level Docker wrapper. Shows how the system injects code into the container via in-memory tar streams and enforces memory/CPU limits so malicious code cannot crash the host.
- **`api/services/boost_service.py`**  
  Extracts live architectural context (`boost:schema` or `model:show`) directly from the Laravel container to feed into the AI's prompt.
- **`docker/laravel-sandbox/Dockerfile`**  
  The recipe for the secure execution environment. It runs Alpine Linux, PHP 8.3, Laravel 12, and Pest 3.

---

## 4. API, Data & Entry Points
These files wire the application together and expose it to the frontend via FastAPI.

- **`api/routers/repair.py`**  
  The main API endpoints (`POST /api/repair`). Crucially, this contains the Server-Sent Events (SSE) logic that streams real-time repair updates back to the frontend UI.
- **`api/main.py`**  
  The FastAPI application startup script. Shows how the database, routers, and background TaskIQ workers are initialized.
- **`api/config.py`**  
  The configuration schema. Shows all the environment variables, API keys, and timeouts the system relies on.
- **`api/models.py`**  
  The SQLAlchemy database schema. Shows exactly what data is persisted for the `Submission` and `Iteration` tracking tables.

---

## 5. System Documentation
- **`MANUAL.md`**  
  The comprehensive system manual. It gives a bird's-eye view of the system architecture, the 13-step loop, deployment steps, and environmental variables.

## Handover Note

**To the Reviewing Engineer:**
When evaluating this codebase, please focus your attention on how the platform handles **non-determinism**. The primary engineering challenge of this project is forcing unpredictable LLM outputs to conform to a hyper-deterministic environment (strict PHP syntax and an 80% mutation score threshold). 

We highly recommend reviewing the **XML patching protocol** in `ai_service.py` and the **Mutation Gate** logic in `sandbox_service.py`, as these are the defining features of the platform's robustness.

Thank you for your time and feedback!
